# Implement Exchange Synchronization for TP/SL Orders

We need to synchronize our local database with the exchange to detect when Take Profit (TP) or Stop Loss (SL) conditional orders are filled by the exchange's matching engine.

## User Review Required
> [!IMPORTANT]
> The proposed approach uses **Periodic Polling** (checking the exchange every 30-60 seconds for open positions) rather than WebSockets. Polling is highly robust, prevents connection drop issues, and doesn't overwhelm the server with multiple persistent connections, which is ideal for this stage of production. Is this acceptable, or do you strictly require millisecond-level WebSocket tracking?

## Open Questions
- Do you want the system to send a **Telegram Notification** automatically when a TP or SL is hit in the background?
- For polling frequency, is checking every 60 seconds acceptable to avoid hitting exchange API rate limits?

## Proposed Changes

### Database Schema Updates
We need to store the order IDs generated by the exchange so we can query their status later.

#### [MODIFY] packages/database/schema.ts
- Add `exchangeOrderId` (text) to the `trades` table to track the main entry order.
- Add `tpOrderId` (text) to the `trades` table to track the Take Profit order.
- Add `slOrderId` (text) to the `trades` table to track the Stop Loss order.

### Execution Engine Updates

#### [MODIFY] apps/engine/src/exchange/ccxt-client.ts
- Capture the response from `exchange.createOrder` for the SL and TP orders.
- Return `order.id`, `tpOrder.id`, and `slOrder.id` back to the server logic.

#### [MODIFY] apps/engine/src/server.ts
- Save the returned `exchangeOrderId`, `tpOrderId`, and `slOrderId` into the database when logging a new trade.
- Integrate a simple periodic worker (e.g., `setInterval`) that runs every 60 seconds.

#### [NEW] apps/engine/src/sync-worker.ts
- Create a background job that:
  1. Queries the database for all `trades` with `status: 'open'` that have a `tpOrderId` or `slOrderId`.
  2. Groups them by `bot.apiKeyId` to minimize API instantiations.
  3. Uses CCXT's `fetchOrder()` to check if the TP or SL orders are marked as `closed` on the exchange.
  4. If a TP or SL is `closed`, it calculates the final PnL based on the actual fill price (`order.average`).
  5. Updates the local `trades` row to `status: 'closed'` and sets the `pnl`.
  6. (Optional) Sends a Telegram alert that the trade was closed via TP/SL.
